## 2. 核心业务逻辑追踪链路 (Execution Traces)

为了实现“指哪打哪”，这里将最复杂的业务逻辑拆解为标准的函数调用链。调试时，请沿着以下链路**依次打断点 (Breakpoints)**。

### 链路 A：AI 聊天核心生命周期
这是本项目最核心的链路，当“发送消息异常”时，按以下顺序排查：
1. **用户触发**：`sendMessage()` (获取输入框内容，校验是否为空)。
2. **构建上下文**：`buildChatContext()` (从 `largeStore` 获取历史消息、系统 Prompt、世界书条目)。
3. **发起请求**：`window.triggerAIResponse(chatId, options)`。
   * **断点点位**：检查 `fetch` 请求的 `body.messages` 是否组装正确。
4. **处理响应 (流式/非流式)**：
   * 成功：解析 JSON 或 Stream Chunk，触发 `appendMessage()` 更新 UI。
   * 失败：触发 `showApiErrorModal(error)`。
5. **持久化**：`largeStore.put('chat_history_' + chatId, history)` 写入数据库。

### 链路 B：大图片安全存储流程
当“图片无法显示”或“上传失败”时：
1. **读取文件**：FileReader 将图片转为 Base64 Data URL。
2. **拦截与转换**：`mediaSaveFromDataUrl(lsKey, dataUrl)`。
   * 核心机制：将 Data URL 转为二进制 `Blob`，存入 IndexedDB (`budingji-media`)，返回 `media:xxx` 字符串。
3. **UI 渲染**：`mediaResolveRef(ref)`。
   * 核心机制：将 `media:xxx` 取出 Blob，使用 `URL.createObjectURL(blob)` 生成浏览器内存链接供 `<img src>` 使用。
   * **注意点**：如果页面刷新，内存链接失效，必须重新走 `mediaResolveRef`。

### 链路 C：后台保活与自动回复
当“应用切后台不工作”时：
1. **入口**：`setInterval(checkBackgroundActivity, 60000)` (每分钟检查一次)。
2. **时间计算**：比对当前时间与 `localStorage.getItem('..._last_trigger')`。
3. **静默请求**：调用 `triggerAIResponse(chatId, { isBackground: true })`。
4. **系统通知**：`budingjiShowSystemNotification()` (若页面不可见，则弹出系统通知)。

## 3. 调试与日志规范 (Debugging & Logging Strategy)

为了在未来 5 分钟内定位问题，后续所有的代码修改与调试必须遵循以下规范：

### 3.1 统一前端控制台日志规范 (Console Tags)
在排查问题时，控制台信息应一目了然。后续代码中推荐使用带有模块 Tag 的日志格式：
*   🟢 **存储层**：`console.log('[Store:IndexedDB] Saved chat history for', chatId)`
*   🔵 **网络层**：`console.info('[API:Fetch] Requesting completion for model', modelName)`
*   🔴 **错误流**：`console.error('[Error:LoveJournal] Failed to parse schedule JSON:', err, rawData)`
*   🟣 **后台任务**：`console.debug('[Background:Timer] Checking activity at', Date.now())`

### 3.2 必备的 DevTools 定位法
如果你需要快速诊断，请遵循此 SOP（标准操作程序）：
1. **遇到 API 报错弹窗**：立即打开 DevTools -> **Network** 面板 -> 筛选 `Fetch/XHR` -> 点击红色的请求 -> 查看 `Preview` 或 `Response` 标签页中的原始报错 JSON。
2. **遇到白屏或数据丢失**：打开 DevTools -> **Application** 面板 -> 展开 `IndexedDB` -> 查看 `budingji_large_store` 和 `budingji-media` 是否有数据。如果没有，说明被浏览器隐私策略清理。
3. **遇到样式错乱**：打开 DevTools -> **Elements** 面板 -> 检查该元素的父级 `.app-modal` 是否缺少 `active` class。

## 4. 深度代码函数索引库 (Deep Function Index)

以下是底层核心函数的绝对定位索引，用于源码级修改时快速 `Ctrl+F` 搜索：

### 📁 `script.js` (系统引擎)
| 函数签名 | 核心作用 | 潜在风险点 / 修改注意 |
| :--- | :--- | :--- |
| `const largeStore = (() => {...})()` | 闭包封装的大文本存储单例 | 内部维护了 `cache` 字典，必须确保 DB 写入与 `cache` 同步，否则会导致读写不一致。 |
| `async function mediaResolveRef(ref)` | 将 `media:id` 转为 Blob URL | 存在 `mediaUrlCache` 内存泄漏风险。长时间不刷新页面可能积累过多 Blob URL。 |
| `function showApiErrorModal(error)` | 拦截所有 API 报错并解析展示 | 当 OpenAI 或第三方 API 返回非标格式的错误体时，解析正则可能失效，需要在此处补充 `try...catch`。 |
| `function runChatIdMigration()` | 将旧版的 RealName 迁移为 UUID | 极其脆弱的历史遗留兼容代码，轻易不要修改。 |
| `async function triggerAIResponse()` | 核心 AI 对话逻辑触发器 | 处理了“流式”与“非流式”的分支。修改 Stream 拼装逻辑时容易引发死循环或 UI 卡顿。 |

### 📁 `love_journal.js` (恋爱志子系统)
| 函数签名 | 核心作用 | 潜在风险点 / 修改注意 |
| :--- | :--- | :--- |
| `async function requestPhoneLockDataFromApi()` | 调用 AI 生成锁屏密码与问题 | 强依赖 AI 的 JSON 输出能力。如果 AI 输出带 markdown 标记 (如 ` ```json `)，需确保正则 `/\{[\s\S]*\}/` 能够正确提取。 |
| `function verifyPasscode()` | 校验用户输入的 4 位密码 | 密码输入采用缓冲字符串 `passcodeBuffer`，失败 3 次后才显示忘记密码按钮。 |
| `function renderSchedule(scheduleDataStr)` | 将 AI 生成的日程 JSON 渲染为 DOM | 高度依赖 DOM 拼接。如果 `scheduleDataStr` 格式损坏，会导致整个页面渲染白屏。此处需加强 fallback 保护。 |

### 📁 `service-worker.js` (PWA 离线支持)
| 函数签名 | 核心作用 | 潜在风险点 / 修改注意 |
| :--- | :--- | :--- |
| `self.addEventListener('fetch', ...)` | 拦截所有网络请求并提供离线缓存 | 目前仅缓存静态资源。绝不能缓存 API 请求 (`/chat/completions`)，否则会导致 AI 回复死循环重复。 |

## 5. 总结：未来解决问题的标准 SOP
如果在未来我（User）报告了一个 Bug（如：“布丁酱机突然不回消息了”）：
1. 你（Agent）首先查阅**第1节映射表**，锁定到 `script.js` 的 `triggerAIResponse`。
2. 指导我提供 DevTools **Network** 面板的报错信息。
3. 查阅**第2节追踪链路**，分析是组装 Context 出错，还是 API 密钥过期。
4. 在 5 分钟内输出明确的代码修改建议或配置调整方案。
